# DCC-EX `myAutomation.h` Generator — Design

**Date:** 2026-04-09
**Status:** Approved (verbal), implementation plan to follow
**Branch:** `io-build-fix`
**Related:** [DCC-EX Stage 2 docs](https://dcc-ex.com/big-picture/stage2.html)

---

## Goal

Generate a `myAutomation.h` file for each `dcc-ex` device in a layout, containing the layout's locomotive roster as EXRAIL `ROSTER(...)` directives. Architect the generator so future EXRAIL features (automations, sequences, routes, etc.) plug in as additional sections without rewriting the core.

**v1 scope:** ROSTER section only.

**v1 non-goals:** SEQUENCE/AUTOMATION/ROUTE blocks, syntax highlighting in the cloud preview, automatic deployment to a DCC-EX sketch folder, diffing against currently-deployed config.

---

## Background

DCC-EX command stations support a Stage 2 feature called EXRAIL, configured via a `myAutomation.h` C header file that the user places inside their `CommandStation-EX/` sketch folder. The file is **compiled and flashed with the sketch** — there is no runtime upload mechanism. The roster command syntax is:

```c
ROSTER(address, "Loco Name", "F0/F1/*F2/F3/Mute/F9//")
```

- `address`: DCC decoder address (1–9999)
- Function labels: slash-separated, positional from F0 upward, empty slot = `//`, `*` prefix = momentary
- The functions string may be `""` for locos with no functions

Our existing `Loco` and `LocoFunction` types in `@repo/modules` already contain everything ROSTER needs (`address`, `name`, `functions[].id`, `functions[].label`, `functions[].isMomentary`). Locos live at `layouts/{layoutId}/locos/{address}` in Firestore — they are layout-wide, not device-bound, so a dcc-ex device emits a roster of every loco in its parent layout.

**Earlier this session,** `dcc-ex` was excluded from `resolvePlatform()` in `io/scripts/lib/bundle.ts` on the rationale that DCC-EX devices have their own firmware sources. That decision is correct for *firmware* generation but blocks *config* generation. This design re-includes `dcc-ex` as its own platform that produces a single config file (`myAutomation.h`) rather than a sketch bundle.

---

## Design Decisions Locked

| # | Decision | Choice |
|---|---|---|
| 1 | Output destination | **A** — Write to `dist/<layoutId>/dcc-ex/<deviceId>/myAutomation.h`. User manually copies/symlinks into their DCC-EX sketch folder. No new Firebase fields. No deploy prompts. |
| 2 | File contents shape | **B** — Section markers (`// === ROSTER (auto-generated by DEJA.js — do not edit between markers) ===` … `// === END ROSTER ===`). No top-of-file provenance header. |
| 3 | Code structure | **B** — File-per-section under `packages/modules/device-config/dcc-ex/`. Orchestrator imports each section and concatenates. |
| 4 | Locos fetch strategy in `io/scripts/lib/firebase.ts` | **3b** — Conditionally fetch locos only when `device.type === 'dcc-ex'` (avoids extra Firestore read for arduino/pico builds). |
| 5 | Cloud preview | **5b** — Add live-reactive preview panel to existing Device Detail Modal in `apps/cloud/src/Connect/Connect.vue` for dcc-ex devices only. |

### Section contract

Every present and future EXRAIL section implements this interface:

```ts
// packages/modules/device-config/dcc-ex/types.ts
import type { Loco, Device } from '@repo/modules'

export interface DccExConfigInput {
  device: Device
  layoutId: string
  locos: Loco[]
}

export interface DccExSection {
  /** Marker name written into the comment delimiters (UPPERCASE convention) */
  name: string
  /**
   * Pure function. Returns the section body (no markers).
   * Returns '' when there's nothing to emit — orchestrator skips empty sections entirely.
   */
  generate(input: DccExConfigInput): string
}
```

### Orchestrator

```ts
// packages/modules/device-config/dcc-ex/index.ts
import { rosterSection } from './roster'
import { wrapSection } from './markers'
import type { DccExConfigInput, DccExSection } from './types'

// Order is significant — sections appear in this order in the output file.
// Adding a new section: import it, add it to this array, ship.
const SECTIONS: DccExSection[] = [
  rosterSection,
  // future: automationSection, sequencesSection, routesSection, ...
]

export function generateDccExAutomation(input: DccExConfigInput): string {
  return SECTIONS
    .map(section => {
      const body = section.generate(input)
      return body ? wrapSection(section.name, body) : ''
    })
    .filter(chunk => chunk !== '')
    .join('\n\n')
}

export type { DccExConfigInput, DccExSection }
```

### Marker helper

```ts
// packages/modules/device-config/dcc-ex/markers.ts
export function wrapSection(name: string, body: string): string {
  if (!body) return ''
  return [
    `// === ${name} (auto-generated by DEJA.js — do not edit between markers) ===`,
    body,
    `// === END ${name} ===`,
  ].join('\n')
}
```

### v1 roster section

```ts
// packages/modules/device-config/dcc-ex/roster.ts
import type { Loco, LocoFunction } from '@repo/modules'
import type { DccExConfigInput, DccExSection } from './types'

export const rosterSection: DccExSection = {
  name: 'ROSTER',
  generate({ locos }: DccExConfigInput): string {
    if (!locos || locos.length === 0) return ''

    const lines = [...locos]
      .sort((a, b) => a.address - b.address)
      .map(formatRosterLine)

    return lines.join('\n')
  },
}

function formatRosterLine(loco: Loco): string {
  const name = escapeCString(loco.name)
  const functionsStr = formatFunctions(loco.functions ?? [])
  return `ROSTER(${loco.address},"${name}","${functionsStr}")`
}

function formatFunctions(functions: LocoFunction[]): string {
  if (functions.length === 0) return ''

  // Sort by id and pad gaps with empty slots
  const sorted = [...functions].sort((a, b) => a.id - b.id)
  const maxId = sorted[sorted.length - 1].id
  const slots: string[] = []

  for (let id = 0; id <= maxId; id++) {
    const fn = sorted.find(f => f.id === id)
    if (!fn) {
      slots.push('') // gap
      continue
    }
    const label = (fn.label && fn.label.trim()) || `F${id}`
    const sanitized = label.replace(/[/\n\r"]/g, ' ').trim()
    slots.push(fn.isMomentary ? `*${sanitized}` : sanitized)
  }

  return slots.join('/')
}

function escapeCString(s: string): string {
  return s
    .replace(/\\/g, '\\\\') // backslashes first
    .replace(/"/g, '\\"')
    .replace(/[\n\r]/g, ' ')
}
```

### Edge case handling (locked from Section 2 review)

| Input | Output |
|---|---|
| Loco with `functions: undefined` or `[]` | `ROSTER(3,"Eng 3","")` |
| Functions out of order `[F2, F0, F1]` | Sorted ascending before emit |
| Function id gaps `[F0, F1, F5]` | `"Light/Bell///Whistle"` (3 empty slots) |
| Function with empty/missing `label` | Falls back to `F${id}` so the throttle has *something* to display |
| `function.isMomentary === true` | `*` prefix on label (`"*Whistle"`) |
| `loco.name` containing `"` | Escaped as `\"` |
| `loco.name` containing `\` | Escaped as `\\` |
| `loco.name` containing `\n` or `\r` | Replaced with space |
| `function.label` containing `/` | Replaced with space (slash is reserved separator) |
| `function.label` containing `"` | Replaced with space (avoids needing per-label escape inside the joined string) |
| Multiple locos | Sorted by `address` ascending in output |
| Calling `generate()` twice with identical input | Byte-identical output (stable sort, no timestamps in output) |

**Stability is non-negotiable.** Re-running `pnpm build` against unchanged Firebase data must produce a byte-identical file so users can `git diff` their `myAutomation.h` and trust the result.

### `io/scripts/lib/firebase.ts` (3b — conditional locos fetch)

```ts
export async function getDeviceConfig(
  layoutId: string,
  deviceId: string
): Promise<DeviceConfigInput> {
  initFirebase()
  const db = getFirestore()
  const layoutRef = db.collection('layouts').doc(layoutId)

  const [deviceSnap, effectsSnap, turnoutsSnap] = await Promise.all([
    layoutRef.collection('devices').doc(deviceId).get(),
    layoutRef.collection('effects').where('device', '==', deviceId).get(),
    layoutRef.collection('turnouts').where('device', '==', deviceId).get(),
  ])

  if (!deviceSnap.exists) {
    throw new Error(`❌ Device "${deviceId}" not found in layout "${layoutId}"`)
  }

  const device = { id: deviceSnap.id, ...deviceSnap.data() } as Device
  const effects = effectsSnap.docs.map(d => ({ id: d.id, ...d.data() }) as Effect)
  const turnouts = turnoutsSnap.docs.map(d => ({ id: d.id, ...d.data() }) as Turnout)

  // dcc-ex devices need the layout-wide loco roster for myAutomation.h generation.
  let locos: Loco[] | undefined
  if (device.type === 'dcc-ex') {
    const locosSnap = await layoutRef.collection('locos').get()
    locos = locosSnap.docs.map(d => ({ id: d.id, ...d.data() }) as Loco)
  }

  return { device, effects, turnouts, locos }
}
```

## Cloud preview (5b)

### Where it lives — match the existing pattern

The cloud app already has a config preview pattern in `apps/cloud/src/Layout/Devices/`:
- **`useDeviceConfig.ts`** is a composable that wraps `generateArduinoConfig` / `generatePicoConfig` / `generatePicoSettings` in `computed()` refs and exposes `downloadPackage()` for ZIP downloads
- **`DeviceDetails.vue`** consumes the composable and renders the previews
- **`DeviceDownload.vue`** consumes the composable for the download workflow

The dcc-ex preview integrates into the **same** pattern:

1. **Extend `useDeviceConfig.ts`** with a new `isDccEx` computed and a new `dccExAutomationH` computed.
2. **Render the preview in `DeviceDetails.vue`** in a section conditional on `isDccEx`, alongside the existing arduino/pico preview blocks.
3. **No new shared component in `@repo/ui`** — the preview UI is local to the cloud app, matching where the arduino/pico previews already live.
4. **Connect.vue's Device Detail Modal stays unchanged** — that modal is the lightweight status view (`<DeviceStatusItem>`), not the full config view. Users reach the full config preview via the existing navigation to `DeviceDetails.vue`.

### Pre-existing bug to fix as part of this work

`useDeviceConfig.ts:20` currently defines:
```ts
const isArduino = computed(() =>
  ['dcc-ex', 'deja-arduino', 'deja-arduino-led'].includes(device.value?.type || '')
)
```

This is wrong — `dcc-ex` is not Arduino. Calling `generateArduinoConfig()` on a dcc-ex device produces nonsensical output (TurnoutPulser arrays, output pins, etc., none of which apply to a DCC-EX command station). This is a pre-existing bug that we adjacent-fix as part of this work since adding dcc-ex support cleanly requires the type-detection logic to be correct.

**Fix:**
```ts
const isArduino = computed(() =>
  ['deja-arduino', 'deja-arduino-led'].includes(device.value?.type || '')
)
const isDccEx = computed(() => device.value?.type === 'dcc-ex')
```

`deja-arduino-led` stays in `isArduino` because it's the LED variant of the arduino sketch and *does* legitimately want config.h preview (even though we exclude it from the io firmware build). That's a separate inconsistency we don't fix here.

### Composable extension

```ts
import {
  generateArduinoConfig,
  generateDccExAutomation,   // ← NEW
  generatePicoConfig,
  generatePicoSettings,
  type Device, type Effect, type Turnout, type Loco,   // ← Loco NEW
} from '@repo/modules'

interface UseDeviceConfigOptions {
  device: Ref<Device | null>
  effects: Ref<Effect[]> | ComputedRef<Effect[]>
  turnouts: Ref<Turnout[]> | ComputedRef<Turnout[]>
  locos?: Ref<Loco[]> | ComputedRef<Loco[]>   // ← NEW, optional (only dcc-ex needs it)
  layoutId?: Ref<string> | ComputedRef<string> // ← NEW, used for context
}

export function useDeviceConfig({ device, effects, turnouts, locos, layoutId }: UseDeviceConfigOptions) {
  const isArduino = computed(() =>
    ['deja-arduino', 'deja-arduino-led'].includes(device.value?.type || '')
  )
  const isPicoW = computed(() => device.value?.type === 'deja-mqtt')
  const isDccEx = computed(() => device.value?.type === 'dcc-ex')   // ← NEW

  // ... existing arduinoConfigH / picoSettingsToml / picoConfigJson unchanged ...

  // 🚂 dcc-ex myAutomation.h
  const dccExAutomationH = computed(() => {
    if (!device.value || !isDccEx.value) return ''
    return generateDccExAutomation({
      device: device.value,
      layoutId: layoutId?.value ?? '',
      locos: locos?.value ?? [],
    })
  })

  // downloadPackage() extended to handle dcc-ex:
  // when isDccEx, the ZIP contains a single myAutomation.h file at the root.

  return {
    isArduino,
    isPicoW,
    isDccEx,         // ← NEW
    arduinoConfigH,
    picoSettingsToml,
    picoConfigJson,
    dccExAutomationH, // ← NEW
    downloadPackage,
  }
}
```
